# 商城管理系统升级需求文档

> 版本：v0.1  
> 项目：vue3_shopManage  
> 目标：从单一商城后台升级为全栈、多端、多角色、多租户系统

## 1. 项目背景

当前项目是一个基于 Vue3、Element Plus、Express、MySQL 的商城后台管理系统，已经具备登录、用户管理、权限管理、商品管理、订单管理、数据报表等基础能力。

后续计划将项目升级为一个更完整的现代化商城 SaaS 系统，支持平台方、租户方、消费者等不同用户群体，并支持 PC 管理后台、移动管理端、商城 H5、小程序等多端使用。

## 2. 项目目标

系统升级后的总体目标是：

> 构建一个支持多租户、多角色、多端访问的商城 SaaS 管理平台。

核心目标包括：

- 建立完整的前后端分离全栈架构。
- 支持平台方和多个租户独立运营。
- 支持多角色权限控制。
- 支持租户数据隔离。
- 支持商城核心业务闭环。
- 支持多端访问和后续扩展。
- 为后续 AI 能力接入预留架构空间。

## 3. 系统定位

升级后的系统定位为：

> 面向平台方和商户的商城 SaaS 管理系统，支持租户入驻、商品管理、订单管理、用户管理、权限管理、数据统计、运营配置和多端访问。

- 租户管理员

## 6. 多租户需求

### 6.1 租户定义

租户是系统中的独立业务主体，可以是一个商家、品牌、公司或组织。

每个租户拥有独立数据，包括：

- 租户用户
- 员工
- 角色
- 权限
- 商品
- 分类
- 订单
- 客户
- 店铺配置
- 营销活动
- 数据报表

### 6.2 租户隔离要求

系统必须保证不同租户之间的数据隔离。

基本要求：

- 租户 A 不能查看租户 B 的数据。
- 租户 A 不能操作租户 B 的商品、订单、用户。
- 平台管理员可以按权限查看全局数据。
- 后端接口必须强制校验 `tenant_id`。
- 前端隐藏菜单不能作为唯一权限控制方式。

### 6.3 租户层级模型

初期建议模型：

```text
Platform
└── Tenant
    └── User / Staff
```

后续可扩展为：

```text
Platform
└── Tenant
    └── Store
        └── Staff
```

### 6.4 租户状态

租户状态建议包括：

- 待审核
- 正常
- 已禁用
- 已过期
- 已删除
- 欠费停用，可选

### 6.5 租户套餐，可选

如果系统后续按 SaaS 商业化运营，可以增加租户套餐能力。

套餐可限制：

- 员工数量
- 商品数量
- 店铺数量
- 存储空间
- API 调用次数
- 可用功能模块
- 数据保留时间


```text

├── 新增商品
```

数据权限建议支持：

- 全部数据

- 当前店铺数据
- 当前部门数据

- 平台管理员：可查看全部租户数据。

## 11. 建议系统架构

### 11.1 前端架构建议

后续可以演进为多应用结构：

```text
apps
├── platform-admin      # 平台管理后台
├── tenant-admin        # 租户管理后台
├── mobile-admin        # 移动管理端
├── shop-h5             # 商城 H5
└── shop-miniapp        # 小程序，可选
```

初期也可以先保持单项目，在当前 Vue3 项目中通过模块、路由和权限区分平台端与租户端。

### 11.2 后端架构建议

后端建议按业务模块重构：

```text
server
├── modules
│   ├── auth
│   ├── user
│   ├── tenant
│   ├── role
│   ├── permission
│   ├── product
│   ├── category
│   ├── order
│   ├── payment
│   ├── after-sale
│   ├── report
│   └── system
├── common
├── config
├── database
├── middleware
└── main
```

```json
    "list": [],
    "pagination": {
      "page": 1,
      "pageSize": 20,
      "total": 100
    }
  }

```

## 18. 待确认问题

后续正式进入设计前，需要确认：

1. 多端具体包括哪些端：PC 管理后台、H5 商城、小程序、App、移动管理端？

4. 是否需要完整消费者商城端：购物车、下单、支付、售后？
5. 是否接入真实支付：微信支付、支付宝、模拟支付？
6. 后端是否继续 Express，还是升级到 NestJS 等框架？
7. 是否引入 TypeScript：前端、后端或全栈 TS？
8. AI 能力是核心功能还是后续增强？

## 19. 总结

本次升级的核心不是在当前后台上简单增加页面，而是将项目升级为真正的商城 SaaS 平台。

推荐建设路线：

```text
基础架构升级
    ↓
用户、租户、权限体系
    ↓
商品、订单、库存核心业务
    ↓
多端建设
    ↓
营销、财务、AI 能力
```

当前项目可以作为 UI 和业务模块参考，但如果目标是全栈、多端、多角色、多租户，建议以重构升级为主，而不是在旧结构上继续堆功能。
