# 多租户设计文档

> 版本：v0.1  
> 范围：商城 SaaS 多租户体系设计

## 1. 设计目标

多租户体系需要保证：

- 多个租户共享同一套系统能力。
- 不同租户业务数据互相隔离。
- 平台管理员可以按权限查看或管理全局数据。
- 租户管理员只能管理本租户数据。
- 后续可以扩展租户套餐、租户到期、租户计费和多店铺。

## 2. 租户定义

租户是系统中的独立业务主体，可以是一个商家、品牌、公司或组织。

每个租户拥有：

- 独立员工
- 独立角色权限
- 独立商品
- 独立订单
- 独立客户
- 独立配置
- 独立数据报表

## 3. 租户层级

第一阶段采用：

```text
Platform
└── Tenant
    └── User / Staff
```

后续扩展为：

```text
Platform
└── Tenant
    └── Store
        └── Staff
```

第一阶段不实现 Store，以降低复杂度。

## 4. 租户状态

租户状态包括：

| 状态 | 说明 |
| --- | --- |
| pending | 待审核 |
| active | 正常 |
| disabled | 已禁用 |
| expired | 已过期 |

后续可扩展：

- deleted
- suspended
- unpaid

## 5. 用户身份类型

系统用户分为：

| 类型 | 说明 |
| --- | --- |
| platform | 平台用户，不隶属于某个租户 |
| tenant | 租户用户，必须绑定 tenant_id |
| customer | 消费者用户，商城前台使用 |

第一阶段优先实现：

- platform
- tenant

## 6. 租户上下文来源

后端租户上下文优先从认证信息中获得：

```text
JWT -> userId -> user.type + user.tenantId
```

平台用户可通过查询参数或 header 指定当前查看的租户上下文，但必须经过权限校验。

租户用户不能手动切换 tenantId。

## 7. API 隔离规则

租户用户请求业务 API 时：

```text
where.tenant_id = currentUser.tenantId
```

平台用户请求业务 API 时：

- 如果具备全局权限，可以查询所有租户数据。
- 如果指定 tenantId，则查询指定租户数据。
- 如果没有全局权限，则拒绝访问。

## 8. 前端租户上下文

前端需要维护：

- 当前用户类型
- 当前用户所属 tenantId
- 当前平台查看 tenantId，可选
- 当前租户状态
- 当前租户套餐信息

建议 Pinia store：

```text
stores/tenant.ts
```

职责：

- 获取当前租户
- 平台用户切换查看租户
- 检查租户状态
- 提供 tenantId 给请求层

## 9. 租户初始化流程

平台创建租户时：

1. 创建 Tenant。
2. 创建租户管理员 User。
3. 创建租户默认角色。
4. 绑定默认权限。
5. 初始化租户设置。
6. 写入操作日志。

## 10. 租户禁用策略

当租户被禁用或过期：

- 租户用户不能登录。
- 已登录租户用户下一次请求应被拒绝。
- 平台用户仍可查看该租户数据。
- 不应直接删除租户业务数据。

## 11. 开发要求

- 禁止前端直接决定租户隔离。
- 禁止客户端传入 tenantId 后后端直接信任。
- 所有租户业务表必须包含 tenant_id。
- 所有租户业务查询必须经过租户上下文处理。
- 所有审计日志必须记录 tenantId。
